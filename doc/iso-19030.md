# ISO 19030 — Framework Explainer

*Context: Speed Loss Dashboard — computing and visualizing speed loss under the ISO 19030 framework, showing propulsion performance trends over time, hull fouling attribution, and correlation with maintenance events.*

---

## What ISO 19030 Actually Is (Scoping First)

Full title: **Ships and marine technology — Measurement of changes in hull and propeller performance** (2016).

Two keywords define its scope:

- **changes** — it measures **change relative to a baseline**, not absolute propulsion efficiency.
- **hull *and* propeller** — it measures the **combined** effect of hull and propeller. The standard itself provides **no method for separating the two**.

The standard has three parts:

| Part | Content | Relevance to you |
|---|---|---|
| 19030-1 | General principles, terminology, measurement principles, uncertainty framework | Definitional layer |
| 19030-2 | **Default Method** — a complete, directly implementable specification | Primary basis for the dashboard |
| 19030-3 | Alternative methods | Permits deviation from the default method (different corrections, filtering, indicators) as long as Part 1 principles are respected |

---

## The Core Quantity: Speed Loss

At **equal delivered power**:

```
% Speed Loss = (V_measured − V_expected) / V_expected × 100
```

- `V_measured` must be **speed through water (STW)**, never SOG — otherwise current contaminates the signal directly.
- `V_expected` comes from the **reference speed–power curve**: the expected speed corresponding to the corrected power at the applicable draught/trim.
- By convention, performance degradation yields a negative value.

## The Data Processing Chain (Part 2, Default Method)

**1. Acquisition**

- Primary parameters (mandatory): STW, delivered power (shaft torque + revolutions), shaft revolutions, fore/aft draught, water depth, relative wind speed/direction, ship heading, timestamp, GPS position, SOG
- Secondary parameters (recommended): rudder angle, seawater temperature/density, air temperature, air pressure
- Sampling rate: **≥ 0.1 Hz** (one sample every 10 seconds), measured continuously

**2. Validation** — Discard out-of-physical-range values, sensor faults, and missing data.

**3. Filtering (the most critical — and most brutal — step)**

Reject data that fails the validity criteria: rudder angle over limit (excludes steering/turning), shallow-water effects, wind force above the limit (default around Beaufort 8), draught/trim outside the range covered by the reference curve, non-steady-state operation (acceleration/deceleration, telegraph changes).

> **Practical impact: 80–95% of raw data is commonly discarded at this step.** The dashboard must treat "number of valid blocks / data coverage" as a first-class displayed metric. Otherwise the indicators look clean while actually resting on very few samples.

**4. Block Averaging** — Default block length is **10 minutes**. Blocks with insufficient valid samples or excessive standard deviation are discarded wholesale.

**5. Correction** — The default method primarily corrects for **wind resistance** (converting the resistance from relative wind into a power correction). Waves, current, and shallow water are handled by **filtering them out**, not by correcting them.

**6. Computation and Aggregation** — Each valid block yields one speed loss value; these are then averaged over the periods defined by each indicator.

## The Four Performance Indicators (Mapping Neatly to Four Views)

Every indicator compares "mean speed loss over an **evaluation period**" against "mean speed loss over a **reference period**":

| Indicator | Question it answers |
|---|---|
| **Dry-docking performance** | How did this docking interval / this coating scheme perform overall? |
| **In-service performance** | How much has performance degraded relative to the post-docking baseline? |
| **Maintenance trigger** | Is it time to clean the hull / polish the propeller? |
| **Maintenance effect** | How much speed did this maintenance event actually recover? |

For the exact definitions of the reference and evaluation periods, go straight to the Part 2 text (I don't want to hand you wrong day-counts). **Recommendation: make the period boundaries configurable parameters rather than hard-coding them** — Part 3 explicitly permits alternative definitions, and you will need to re-run history.

## Mapping Back to Your Dashboard Spec

**1. "Propulsion performance trends over time"** → Direct mapping. x = block timestamp, y = % speed loss, overlaid with a rolling average and a trend line. The slope of that trend (%/month, or %/1000 days) *is* the fouling rate.

**2. "Hull fouling attribution"** → ⚠️ **This falls outside the scope of ISO 19030.** The standard measures the *combined* hull-plus-propeller change and offers no decomposition. To attribute, you must add an **inference layer** on top of ISO:

- **Slope vs. step change**: fouling is a slow, near-linear degradation (slope); hull cleaning / propeller polishing produces a step recovery.
- **Event correlation**: a step following a standalone propeller polish → attributable to the propeller. A step following a standalone hull clean → the hull.
- **External features**: idle days at anchor, water temperature, biogeographic region of the route, coating type, fraction of time spent at low speed.

Label attribution explicitly in the UI as an **inference layer**, with a confidence level attached. Do not present it as a direct ISO output — this matters a great deal once the output reaches owners/charterers and enters contractual dispute.

**3. "Correlation with maintenance events"** → Maps to Maintenance effect / Maintenance trigger. Events (dry-docking, hull cleaning, propeller polishing, recoating) must be first-class entities in the data model.

> **Watch out:** calibrating or replacing the **speed log itself produces a step change**, and it is very easily misread as "the hull clean worked." This is the single most common false signal in practice.

## Uncertainties You Must Confront Honestly

- **STW is the dominant error source.** Bias and drift in electromagnetic/Doppler speed logs, compounded by the fact that the sensor itself fouls. ISO 19030's overall uncertainty is essentially bounded by speed-log quality.
- **Reference curve quality determines everything.** Is it from sea trials (corrected per ISO 15016), or model tests / CFD? Does it cover a wide enough draught/trim range?
- **The default method does not correct for waves.** It only excludes them indirectly via wind-speed filtering — but swell is not synchronous with local wind.
- **The shorter the evaluation period, the larger the uncertainty.** **Maintenance trigger is intrinsically the noisiest of the four indicators.** Don't let the UI make it look more certain than it is.

---

## Architecture Recommendations

Layer the data model and preserve traceability end-to-end:

```
raw → validated → filtered → blocks → corrected → speed_loss → KPI
```

Let the user drill down from any KPI: which blocks were included, which were filtered out, and **why they were filtered out**. This **auditability** is the core value of an ISO 19030 dashboard — far more so than a pretty line chart.

In parallel, parameterize and version-control filter thresholds, period definitions, and the reference curve version.
